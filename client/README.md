# client

> React 19 SPA for HackerNews — file-based routing, typed API client, and optimistic cache updates.

## Tech Stack

| Layer | Technology |
|---|---|
| Framework | React 19 |
| Routing | TanStack Router (file-based, auto code-split) |
| Server State | TanStack Query v5 (infinite queries, suspense) |
| Forms | TanStack Form + Zod (shared validators from `shared/`) |
| Styling | Tailwind CSS v4 + Radix UI primitives |
| API Client | Hono RPC (`hc<AppType>`) — fully typed from server route definitions |
| Auth | better-auth React client |
| Build | Vite 6 + Bun |

## Getting Started

```bash
bun install
bun run dev
```

The dev server starts at `http://localhost:5173` and proxies `/api` requests to the backend (default `http://localhost:3000`).

## Architecture

### Routing

TanStack Router with file-based route generation. Routes live in `src/routes/` and the route tree is auto-generated by the `@tanstack/router-plugin/vite` plugin.

```
/                  → index.tsx           (posts list)
/post/:id          → post.$id.tsx        (post detail + comments)
/login             → login.tsx
/signup            → signup.tsx
/submit            → _auth/submit.tsx    (protected)
```

The `_auth.tsx` layout route acts as an auth guard — it checks for a current user in `beforeLoad` and redirects to `/login` if unauthenticated. Any route nested under `_auth/` inherits this protection.

Route search params are validated with Zod schemas (`validators/search.validation.ts`). The `searchSchema` is shared between the home page and post detail page for `sortBy` and `order` params.

### API Client

The client uses Hono's `hc` to create a fully typed RPC client from the server's `AppType`:

```ts
import type { AppType } from "@server/index";
import { hc } from "hono/client";

export const api = hc<AppType>(SERVER_URL, {
  fetch: (input, init) => fetch(input, { ...init, credentials: "include" }),
}).api;
```

This gives you autocomplete and type checking on every API call — `api.posts.$get(...)`, `api.posts[":id"].comment.$post(...)`, etc. No manual type wiring or code generation needed.

### Data Fetching

Services in `src/services/` define query option factories consumed by routes and components:

- **`postsInfiniteQueryOptions`** — paginated posts list with sort/filter params
- **`postQueryOptions`** — single post by ID
- **`commentsInfiniteQueryOptions`** — paginated top-level comments for a post
- **`subCommentsInfiniteQueryOptions`** — paginated sub-comments with `initialData` from parent comment's `childComments`

Routes use `loader` functions to prefetch data via `queryClient.ensureQueryData()` / `ensureInfiniteQueryData()` before rendering. Components then read with `useSuspenseQuery` / `useSuspenseInfiniteQuery` for zero-loading-state rendering.

### Optimistic Updates

The `createOptimisticUpdateMutation` factory (`hooks/use-optimistic-update.ts`) is a reusable abstraction for optimistic cache mutations. It takes:

- **`CacheAdapter[]`** — each adapter knows how to `read`, `write`, `cancel`, and `invalidate` a specific cache location (e.g., post detail cache vs. posts list cache)
- **`getOptimisticUpdate`** — pure function that computes the optimistic state from current state
- **`getId`** — deduplication key for concurrent mutations on the same entity

It handles the full lifecycle: cancel in-flight queries → snapshot previous state → apply optimistic update → reconcile with server response or rollback on error. A `pendingMap` tracks concurrent mutations so rapid-fire actions (e.g., toggling upvote quickly) don't lose state.

Used by `useUpvotePost` and `useUpvoteComment`, each defining their own `CacheAdapter` implementations for their respective cache shapes (infinite query pages, detail queries).

### Auth

Authentication is handled by better-auth's React client. The `CurrentUserProvider` context wraps the app and exposes `{ currentUser, isLoading }` via `useCurrentUser()`. The current user is prefetched in `__root.tsx`'s `beforeLoad` so it's available before any route renders.

### Code Splitting

Vite's build config splits vendor dependencies into separate cacheable chunks:

| Chunk | Contents |
|---|---|
| `react-vendor` | react, react-dom |
| `tanstack-query` | @tanstack/react-query |
| `tanstack-router` | @tanstack/react-router |
| `tanstack-form` | @tanstack/react-form |
| `radix-ui` | All @radix-ui packages |
| `ui-vendor` | lucide-react, sonner, cva, clsx, tailwind-merge |
| `better-auth` | better-auth |

TanStack Router's `autoCodeSplitting` handles per-route code splitting automatically. Devtools are lazy-loaded and excluded from production builds.
